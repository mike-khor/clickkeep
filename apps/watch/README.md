# apps/watch — Apple Watch SwiftUI app (stub)

This directory is intentionally a stub. The full plan lives in
[`docs/MOBILE.md`](../../docs/MOBILE.md) (§2 and §3). When Mike is ready to
bootstrap the Watch target, follow the steps in **Bootstrap** below.

The Watch app is a **separate SwiftUI target** in the same Xcode workspace
generated by `apps/ios/` (Capacitor). Capacitor itself never sees the Watch
code; we just piggyback on the iOS Xcode project as the host for the Watch
extension.

## Bootstrap (when ready — not yet)

1. Open the iOS workspace:
   ```bash
   cd apps/ios && npx cap open ios
   ```
2. In Xcode: **File > New > Target…**
   - Choose **watchOS > App**.
   - Product name: `ClickKeepWatch`.
   - Interface: **SwiftUI**. Lifecycle: **SwiftUI App**.
   - Tick **Include Notification Scene**: **off** (we don't use push).
   - Tick **Include Complication**: optional (cute for v2).
   - Click Finish; Xcode adds the target to the workspace.
3. Move the new target's source into this directory and add it as a
   workspace folder reference, so the Swift sources live under
   `apps/watch/App/` in git. (Xcode default puts it in
   `apps/ios/ios/App/ClickKeepWatch Watch App/` — drag it over and update
   the file references.) This keeps the watch code reviewable separately
   from the Capacitor-generated noise.

## Planned architecture

The Watch never speaks to the Durable Object directly in v1 (see
`docs/MOBILE.md` §2 for why). All state flows: **Worker → iPhone (Capacitor)
→ Watch (WatchConnectivity) → BeatScheduler → WKInterfaceDevice**.

### `WatchSessionModel.swift` — the core type

Conforms to `ObservableObject` and `WCSessionDelegate`. Holds:

- `@Published var tempo: [TempoSegment]` — same shape as the TS type
- `@Published var anchorServerTime: Double` — ms since epoch, server clock
- `@Published var clockOffsetMs: Double` — relayed from phone
- `@Published var flash: Bool` — toggles once per beat for the UI dot
- `@Published var bpmLabel: String`, `isPlaying: Bool`, `statusLine: String`

Receives one message type from the phone over `WCSession.sendMessage`:

```swift
struct WatchSyncPayload: Codable {
  let tempo: [TempoSegment]      // mirrors TempoSegment from sync-core/types.ts
  let anchorServerTime: Double   // ms-since-epoch when playback started
  let clockOffsetMs: Double      // add to Date().timeIntervalSince1970 * 1000
  let isPlaying: Bool
}
```

On every payload, the model resets `BeatScheduler` with the new tempo
and offset.

### `BeatScheduler.swift` — Swift port of the click engine

Mirrors `packages/click-engine/src/scheduler.ts` exactly, including the
`lookaheadSec = 0.1` and `scheduleIntervalMs = 25` constants.

```swift
final class BeatScheduler {
  private var timer: DispatchSourceTimer?
  private var nextBeat: Int = 0
  private let onBeat: (Int) -> Void   // beat index — model decides haptic style

  func start(tempo: [TempoSegment], nowServerMs: @escaping () -> Double) {
    let q = DispatchQueue(label: "clickkeep.watch.scheduler")
    let t = DispatchSource.makeTimerSource(queue: q)
    t.schedule(deadline: .now(), repeating: .milliseconds(25))
    t.setEventHandler { [weak self] in
      self?.tick(tempo: tempo, nowServerMs: nowServerMs())
    }
    timer = t
    t.resume()
  }

  private func tick(tempo: [TempoSegment], nowServerMs: Double) {
    let horizon = nowServerMs + 100  // lookaheadSec * 1000
    while true {
      let beatServer = SyncMath.serverTimeForBeat(tempo: tempo, beat: nextBeat)
      if beatServer > horizon { break }
      let delaySec = max(0, (beatServer - nowServerMs) / 1000)
      DispatchQueue.main.asyncAfter(deadline: .now() + delaySec) { [nb = nextBeat] in
        self.onBeat(nb)
      }
      nextBeat += 1
    }
  }

  func stop() { timer?.cancel(); timer = nil }
}
```

### Haptic firing — the actual on-wrist click

In `WatchSessionModel`, the `BeatScheduler`'s `onBeat` callback maps to:

```swift
import WatchKit

private func playHaptic(forBeat beat: Int) {
  let device = WKInterfaceDevice.current()
  let isDownbeat = beat % 4 == 0   // beatsPerBar from tempo segment in v2
  device.play(isDownbeat ? .start : .click)
  // also toggle the UI flash
  Task { @MainActor in self.flash.toggle() }
}
```

`WKHapticType` mapping (mirrors `docs/MOBILE.md` §3 table):

| Beat type | Haptic |
| --- | --- |
| Downbeat | `.start` |
| Ordinary beat | `.click` |
| Song end | `.stop` |
| Sync failure | `.failure` |

### `SyncMath.swift` — Swift mirror of `sync-core/tempo.ts`

The math is short. Manual port; **must be unit-tested** with the same
fixtures as `packages/sync-core/src/tempo.test.ts` so the two
implementations cannot drift. The port is a Tier 2 edit — it doesn't
touch the TypeScript source, but it's a peer implementation of a Tier 3
file, so the test parity is what makes it safe.

## What we will NOT do here

- No **independent WebSocket** from the Watch in v1. The phone is the
  source of truth. (Tracked as a future BACKLOG item under "Watch solo
  mode".)
- No **Complication** on v1. Cute but not load-bearing.
- No **always-on display** customization. The default behavior is fine.
- No **HealthKit** integration. We are a metronome, not a fitness app.
